# EGoclaw Demo 技术执行文档

## 1. 当前 Demo 的真实目标

当前要跑通的不是“抖音账号同步”和“全量行为分析”，而是一条更聚焦、更可落地的 Demo 主链路：

`本地已下载的收藏夹视频 -> 视频内容提取 -> skill 蒸馏与存储 -> 技能图谱构建 -> 抖音网页端问题识别 -> 技能匹配 -> 桌宠提醒 -> 用户确认 -> 进入 app 做详细学习规划`

### 1.1 Demo 输入

当前唯一可靠输入：

1. 本地已经下载好的抖音收藏夹视频
2. 用户在抖音网页端输入的学习问题或搜索问题

### 1.2 Demo 输出

必须产出的结果：

1. 从本地视频中提取可用内容信息
2. 将视频蒸馏成结构化 skill 并存储到技能库
3. 构建技能图谱，明确大技能、小步骤和前置关系
4. 识别用户在抖音网页端输入的问题
5. 从技能库中匹配最相关 skill 和对应视频
6. 由桌宠主动提醒用户
7. 用户确认后跳转到 app 内查看详细掌握规划

### 1.3 当前不做

以下内容不是当前 Demo 主链路，不能继续写成前置依赖：

1. 抖音账号登录与绑定
2. 收藏夹在线同步
3. 点赞、停留、反复观看等行为数据同步
4. 复杂实时流式系统
5. 多 Agent 编排本身作为目标
6. 图数据库或大规模推荐系统

结论：

当前 Demo 应采用“离线预处理 + 在线匹配触发”的单机方案，而不是“账号同步 + 实时画像更新”的平台级方案。

---

## 2. Demo 验收标准

只要满足以下条件，就算 Demo 跑通：

1. 指定本地视频目录后，系统能扫描并处理一批收藏夹视频
2. 每条视频至少能抽取出标题、字幕或 OCR 中的一部分有效信息
3. 系统能从多条视频中蒸馏出一组 skill，并持久化保存
4. 每个 skill 都能拆成若干可学习的小步骤
5. 系统能生成一份可视化或结构化的技能图谱
6. 当用户在抖音网页端输入问题时，系统能自动识别并触发匹配
7. 匹配结果能返回：
   - 对应 skill
   - 对应的支撑视频
   - 推荐先学哪一步
8. 桌宠能弹出提醒并询问是否生成详细规划
9. 用户确认后，app 能打开对应 skill 的详细学习规划页

## 3. 端到端主链路

### 3.1 离线预处理链路

`本地视频目录 -> 视频扫描 -> 元数据提取 -> ASR/OCR/摘要 -> 单视频 skill 蒸馏 -> skill 聚合 -> 技能图谱构建 -> 技能库存储`

### 3.2 在线触发链路

`抖音网页端输入问题 -> 输入监听 -> 查询理解 -> 技能匹配 -> 桌宠提醒 -> 用户确认 -> app 打开详细规划`

### 3.3 App 内继续链路

`加载已匹配 skill -> 展示技能图谱位置 -> 展示掌握步骤 -> 关联推荐视频 -> 生成详细学习计划`

---

## 4. 系统设计原则

### 4.1 架构原则

当前版本优先采用“模块化单体”而不是“大型 MAS 系统”。

理由：

1. 当前输入源固定，本地视频目录即可
2. 当前目标是跑通 Demo，而不是构建平台级基础设施
3. skill 蒸馏、图谱生成、查询匹配都更适合先做成清晰的串行模块
4. 先把关键资产沉淀下来，比先搭复杂 agent 编排更重要

### 4.2 数据原则

当前版本优先沉淀三类核心资产：

1. 视频理解结果
2. 结构化 skill
3. 技能图谱

### 4.3 交互原则

桌宠不是独立做理解的大脑，只负责：

1. 接收匹配结果
2. 触发提醒
3. 收集用户确认或拒绝
4. 把用户带入 app 内继续完成学习规划

## 5. 核心模块

### 5.1 Local Video Ingestion

输入：

1. 本地视频目录
2. 可选的视频清单文件

输出：

1. 标准化视频记录
2. 待处理任务列表

职责：

1. 扫描本地视频文件
2. 生成稳定的 `video_id`
3. 去重
4. 记录文件路径、时长、大小、创建时间

### 5.2 Video Extraction

输入：

1. 本地视频文件

输出：

1. 字幕转写结果
2. 关键帧 OCR 结果
3. 视频摘要
4. 可选的标题或文件名解析结果

职责：

1. 提取语音文本
2. 提取画面中的文字
3. 合并为可供蒸馏的文本上下文

### 5.3 Skill Distillation

输入：

1. 单条视频的综合文本上下文

输出：

1. 视频级 skill 候选
2. 视频教的是什么
3. 面向什么问题
4. 适合什么阶段
5. 从哪一步开始

职责：

1. 将视频理解成“可学习内容”
2. 输出统一 schema，便于后续聚合

### 5.4 Skill Aggregation

输入：

1. 多条视频的蒸馏结果

输出：

1. 去重后的 skill
2. skill 与视频的映射关系

职责：

1. 合并表达不同但实质相同的 skill
2. 为每个 skill 找到支撑视频集合
3. 形成大技能与子技能候选

### 5.5 Skill Graph Builder

输入：

1. 已聚合的 skill
2. 每个 skill 的子步骤

输出：

1. 技能节点
2. 技能关系边
3. 学习顺序建议

职责：

1. 将大技能拆成更小的掌握步骤
2. 标记前置关系、并列关系、进阶关系
3. 输出给前端成长地图使用

### 5.6 Skill Repository

输入：

1. 最终 skill
2. 技能图谱
3. 视频映射信息

输出：

1. 本地技能库文件
2. 可检索的索引

职责：

1. 存储结构化 skill
2. 支持查询匹配
3. 支持 app 页面读取

### 5.7 Douyin Query Listener

输入：

1. 抖音网页端的输入框内容
2. 用户提交或停顿后的查询事件

输出：

1. `query_event`

职责：

1. 监听用户在抖音网页端输入的问题
2. 在合适时机触发查询识别

当前建议实现方式：

1. 浏览器插件 content script
2. 或网页注入脚本

注意：

当前只需要捕获“用户主动输入的问题”，不需要同步账号数据。

### 5.8 Query Matcher

输入：

1. 用户问题
2. 技能库
3. 技能图谱
4. 视频证据

输出：

1. Top skill
2. Top 支撑视频
3. 推荐起步步骤
4. 匹配置信度

职责：

1. 理解用户想学什么
2. 在技能库中找到最相关的 skill
3. 给出“为什么匹配到这个 skill”的证据

### 5.9 Nudge Orchestrator

输入：

1. 匹配结果
2. skill 简述
3. 推荐第一步

输出：

1. 桌宠提醒事件

职责：

1. 生成提醒文案
2. 控制提醒优先级
3. 决定是否打开 app 深链

### 5.10 Learning Planner

输入：

1. 用户确认的 skill
2. 技能图谱
3. skill 的步骤与支撑视频

输出：

1. 详细学习规划
2. 分阶段学习步骤
3. 每一步对应的视频与行动建议

职责：

1. 把“大技能”展开成可掌握的小步骤
2. 输出适合 app 展示的学习规划

## 7. 离线预处理流程

### 7.1 扫描与建档

步骤：

1. 扫描本地视频目录
2. 为每个视频生成 `video_id`
3. 记录基础元数据
4. 判断是否已处理过

### 7.2 内容提取

步骤：

1. 抽音频做字幕转写
2. 抽关键帧做 OCR
3. 合并文件名、标题、字幕、OCR 为统一文本
4. 生成简短摘要

### 7.3 单视频 skill 蒸馏

每条视频至少要回答以下问题：

1. 这个视频在教什么
2. 这个内容属于哪个更大的技能方向
3. 它解决什么问题
4. 更适合哪个阶段的人
5. 从哪一步开始练

### 7.4 多视频聚合

目标：

1. 合并同类 skill
2. 把多个相似视频归到同一个 skill 下
3. 找到大技能与小技能关系

### 7.5 技能图谱生成

输出内容：

1. 大技能节点
2. 子技能节点
3. 掌握步骤节点
4. 节点之间的先后依赖关系

重点：

图谱不只是“内容分类”，而是“学习拆解”。

也就是说：

1. 一个大的 skill 要能拆成几个更小的可掌握步骤
2. 每一步都最好能落到一个明确动作
3. 每一步都尽量关联一个或多个支撑视频

## 8. 在线查询匹配流程

### 8.1 触发时机

当前版本只在以下场景触发：

1. 用户在抖音网页端输入问题
2. 用户提交搜索或输入停顿达到阈值

当前不做：

1. 长时间持续监听页面全部行为
2. 未经触发的主动猜测式推荐

### 8.2 查询理解

对用户输入的问题做标准化处理：

1. 分词
2. 同义词归一
3. 问题场景识别
4. 学习意图识别

例如：

- “怎么练面试自我介绍”
- “面试开场怎么说”
- “自我介绍总是讲乱怎么办”

都可能落到同一 skill：`面试自我介绍`

### 8.3 匹配策略

当前建议使用“关键词召回 + 语义相似度 + 图谱加权”的混合策略。

参考公式：

`match_score = 0.45 * semantic_similarity + 0.25 * keyword_overlap + 0.20 * scenario_match + 0.10 * graph_priority`

其中：

1. `semantic_similarity`：query 与 skill 描述、步骤描述的语义相似度
2. `keyword_overlap`：query 与 skill 名称、别名的关键词重合度
3. `scenario_match`：query 是否命中 skill 的问题场景
4. `graph_priority`：优先选择图谱中更完整、步骤更清晰、视频支撑更多的 skill

### 8.4 匹配输出要求

每次匹配至少输出：

1. Top 1 skill
2. Top 1 推荐起步步骤
3. Top 1 到 Top 3 支撑视频
4. 匹配理由

如果没有足够可信的匹配：

1. 不触发桌宠提醒
2. 只记录查询事件，等待后续扩充技能库

## 9. 桌宠提醒与 app 转场

### 9.1 桌宠职责

桌宠只做两件事：

1. 告诉用户“你的问题已经匹配到技能”
2. 询问用户“是否需要进一步做详细掌握规划”

### 9.2 桌宠提醒文案模板

模板：

`你刚刚在抖音里搜了“{query}”，我已经帮你找到对应技能“{skill_name}”和相关视频，要不要我继续帮你拆成详细掌握步骤？`

### 9.3 用户操作

桌宠至少提供 3 个动作：

1. `开始规划`
2. `先看看技能图谱`
3. `先不用`

### 9.4 转入 app

用户点击 `开始规划` 后：

1. 打开 app 对应 skill 页面
2. 带上 `skill_id`
3. 带上 `query_text`
4. 带上推荐起步步骤

app 内展示：

1. 当前 skill 卡片
2. 该 skill 在整张技能图谱中的位置
3. 该 skill 的掌握步骤
4. 每一步关联的视频证据
5. 详细学习规划

## 12. 当前开发顺序

### 12.1 第一优先级

必须先做：

1. 本地视频扫描与建档
2. 视频字幕/OCR/摘要抽取
3. 单视频 skill 蒸馏 schema
4. 多视频 skill 聚合
5. 技能图谱 schema
6. skill 库持久化

### 12.2 第二优先级

随后做：

1. 抖音网页端输入监听
2. 查询匹配服务
3. 桌宠提醒事件
4. app 内 skill 规划页

### 12.3 暂不做

当前不要继续投入：

1. 抖音账号同步
2. 用户行为画像系统
3. 实时增量同步
4. 多 Agent 编排框架
5. 复杂图数据库

## 13. 风险与降级方案

### 13.1 视频提取质量不足

问题：

1. 字幕转写不准
2. OCR 提取不完整

降级：

1. 允许人工补充标题或摘要
2. 允许先用文件名和人工标签做冷启动

### 13.2 skill 聚合不稳定

问题：

1. 相似 skill 合并错
2. 大技能拆解不合理

降级：

1. 先人工审核 Top skill
2. 先从少量高频主题做固定模板

### 13.3 抖音网页端输入监听不稳定

问题：

1. 页面结构变化
2. 输入框选择器失效

降级：

1. 增加手动触发按钮
2. 增加独立测试输入框验证匹配链路

### 13.4 桌宠系统集成复杂

问题：

1. 真正系统级桌宠实现成本高

降级：

1. 第一版先用前端悬浮灵宝
2. 只验证提醒和转场，不追求系统级桌面能力

## 14. 当前版本的最终结论

当前 Demo 的正确技术方向不是：

`抖音账号同步 -> 行为意图分析 -> 动作推荐`

而是：

`本地收藏夹视频 -> skill 蒸馏 -> 技能图谱 -> 抖音网页端问题识别 -> 技能匹配 -> 桌宠提醒 -> app 内详细规划`

因此，接下来所有实现和文档都应围绕这 4 个核心资产展开：

1. 本地视频理解结果
2. skill 技能库
3. 技能图谱
4. 查询匹配与桌宠转场

只要这条链路跑通，Demo 就成立。
