# EGoclaw 技术执行文档

## 1. 目标系统

目标输入：

1. 用户登录后同步抖音收藏夹
2. 用户登录后同步抖音行为痕迹
3. 用户后续新增收藏和行为增量

目标输出：

1. 实时或准实时更新用户意图
2. 生成当前最值得开始的下一步
3. 由灵宝以桌宠形态主动提醒

主链路：

`抖音数据同步 -> 内容蒸馏 -> 意图识别 -> 图谱更新 -> 动作规划 -> 桌宠提醒`

系统分层：

1. 软件层
2. 交互层
3. 事件总线

软件层负责：

1. 同步抖音数据
2. 蒸馏视频内容
3. 构建个人知识图谱
4. 识别用户意图
5. 规划下一步动作
6. 判断是否提醒

交互层负责：

1. 以灵宝桌宠形态承接系统输出
2. 展示气泡文案、动作卡片、成长地图
3. 接收用户点击、忽略、延后、开始等反馈

事件总线负责：

1. 将软件层判断结果推送给桌宠
2. 将桌宠交互反馈回写给软件层

## 2. 当前可执行路线

### Phase 1: Demo

现在做的事：

1. 用 Mock 收藏数据搭一条完整链路
2. 跑通视频蒸馏、意图识别、动作规划、提醒生成
3. 做一个桌宠式 Demo 界面

验收标准：

1. 输入 20 到 50 条 mock 视频
2. 输出 3 个以内主要意图
3. 每个意图给 1 个可执行动作
4. 灵宝能生成对应提醒

### Phase 2: 同步链路验证

现在必须并行验证的事：

1. 抖音账号登录能否建立连接
2. 收藏夹能否同步
3. 行为数据能否同步
4. 是否支持增量更新
5. 是否满足合规要求

结论标准：

1. 上述 5 项成立，账号同步可作为 MVP 输入
2. 任一项不成立，MVP 不能依赖账号同步

### Phase 3: MVP

如果同步链路成立：

1. 登录
2. 同步收藏夹
3. 进入分析链路
4. 生成意图和动作
5. 桌宠提醒

如果同步链路不成立：

1. 使用过渡性导入方案
2. 保留后续同步链路验证

## 3. MAS 系统

### 3.1 Agent 列表

#### Ingestion Agent

输入：

- 收藏夹条目
- 行为事件
- mock 数据

输出：

- 标准化内容对象
- 待处理任务

职责：

- 去重
- 标准化
- 分发

#### Video Understanding Agent

输入：

- 标题
- 描述
- OCR
- ASR
- 作者信息

输出：

- topic
- skill
- stage
- goal
- friction
- recommended_action
- confidence

职责：

- 单视频语义蒸馏

#### Intent Analyzer Agent

输入：

- 多条视频蒸馏结果
- 用户近期行为
- 用户历史反馈

输出：

- intent_id
- intent_name
- score
- confidence
- evidence
- supporting_videos

职责：

- 识别当前最强意图
- 做实时或准实时更新

#### Profile Agent

输入：

- 历史意图
- 行为反馈
- 动作执行记录

输出：

- 长期兴趣分布
- 偏好开始方式
- 中断模式
- 动作强度偏好

职责：

- 维护长期用户画像

#### Graph Builder Agent

输入：

- 视频蒸馏结果
- 意图结果
- 动作结果

输出：

- 图谱节点和关系更新

职责：

- 维护长期记忆层

#### Planner Agent

输入：

- 当前意图
- 用户画像
- 图谱关系

输出：

- 当前最优动作
- why_now
- estimated_minutes

职责：

- 选当前该推的一步

#### Nudge Agent

输入：

- 当前动作
- 当前意图
- 触发原因

输出：

- 提醒文案
- 提醒类型
- 触发时机

职责：

- 驱动灵宝提醒

#### Review Agent

输入：

- 点击
- 忽略
- 完成

输出：

- 意图权重修正
- 画像修正
- 后续策略修正

职责：

- 反馈闭环

### 3.2 MAS 执行流程

流程 1：初始化分析

1. Ingestion Agent 接收收藏数据
2. Video Understanding Agent 逐条蒸馏
3. Intent Analyzer Agent 生成当前意图
4. Graph Builder Agent 写入图谱
5. Planner Agent 生成第一步
6. Nudge Agent 生成提醒

流程 2：增量更新

1. 接收新增收藏或行为事件
2. 更新对应视频和行为记录
3. 重算意图分数
4. 更新当前推荐动作
5. 触发或跳过提醒

流程 3：反馈闭环

1. 用户点击、忽略或完成动作
2. Review Agent 更新画像和意图
3. Planner Agent 更新下一步

### 3.3 软件层与交互层的连接

软件层不直接控制桌宠 UI，只输出结构化事件。

桌宠只接收以下事件：

1. `nudge_ready`
2. `intent_changed`
3. `action_completed`

事件结构示例：

```json
{
  "event_type": "nudge_ready",
  "user_id": "u_001",
  "intent_id": "interview_expression",
  "action_text": "今晚练1分钟自我介绍",
  "trigger_reason": [
    "近7天重复收藏面试表达内容",
    "今晚为空闲时段"
  ],
  "message_type": "gentle_prompt",
  "priority": "high"
}
```

桌宠反馈事件：

1. `pet_action_clicked`
2. `pet_dismissed`
3. `pet_snoozed`
4. `action_started`

反馈示例：

```json
{
  "event_type": "pet_snoozed",
  "user_id": "u_001",
  "intent_id": "interview_expression",
  "snooze_minutes": 120
}
```

### 3.4 意图分析方法

输入信号：

1. 收藏频次
2. 最近时间窗口重复出现主题
3. 反复观看或停留
4. 历史动作执行情况
5. 用户对提醒的反馈

意图分数：

`intent_score = recency + frequency + repeat_behavior + action_conversion + feedback_weight`

输出要求：

1. 每次只保留 Top 3 意图
2. 每个意图必须带 evidence
3. 每个意图必须能映射到一个具体动作

### 3.5 提醒触发逻辑

提醒不是定时弹窗，只在满足上下文条件时生成 `nudge_ready`。

触发条件：

1. 当前存在明确 Top intent
2. 当前 intent 已映射到具体 action
3. 用户近期未连续忽略同类提醒
4. 当前时间窗口适合触发
5. 当前触发原因明确

触发原因可以包括：

1. 今晚有空闲时间
2. 连续几天出现同类主题
3. 之前开始过但中断
4. 最近重新出现同类内容
5. 临近面试、汇报、减脂、旅行等场景

触发结果结构：

```json
{
  "should_nudge": true,
  "intent_id": "ppt_beginner",
  "action_text": "今晚做1页PPT开场页",
  "priority": "medium",
  "reason": [
    "近5天重复出现PPT内容",
    "今晚20:00-22:00为空闲时段"
  ]
}
```

### 3.6 桌宠运行层

桌宠不做内容理解，只负责承接系统输出并完成交互。

桌宠运行状态：

1. `idle`
2. `appearing`
3. `speaking`
4. `waiting_response`
5. `guiding`
6. `sleeping`

桌宠支持的动作：

1. 弹出提醒气泡
2. 展示行动卡片
3. 打开成长地图
4. 延后提醒
5. 记录忽略

桌宠文案由 `Nudge Agent` 输出，桌宠前端只负责渲染。

### 3.7 第一版桌宠实现

第一版不做完整系统级桌面宠物，先做前端悬浮式桌宠。

第一版实现：

1. Web 页面右下角悬浮灵宝
2. 接收到 `nudge_ready` 后弹出气泡
3. 用户可点击：
   - 现在开始
   - 晚点提醒
   - 不是这个
4. 点击后进入成长地图或当前动作卡片

验证点：

1. 用户是否愿意点开提醒
2. 用户是否理解桌宠价值
3. 提醒是否被认为自然而非打扰

## 4. 数据获取方案

### 4.1 主方案：账号同步

需要验证：

1. 是否能登录并绑定抖音账号
2. 是否能同步收藏夹
3. 是否能同步行为痕迹
4. 是否能做增量同步
5. 是否合规

只有 5 项全部成立，账号同步才能进入 MVP。

### 4.2 过渡方案

如果账号同步不成立，当前可执行输入只保留：

1. 抖音链接导入
2. 收藏页截图导入
3. 视频列表批量导入

作用：

1. 跑通蒸馏
2. 跑通意图识别
3. 跑通图谱
4. 跑通桌宠提醒

### 4.3 半自动方案

可做方向：

1. 浏览器插件读取当前页面收藏项
2. 分享到 EGoclaw
3. 本地导出记录解析

用途：

1. 降低输入成本
2. 提高样本量

### 4.4 非官方采集方案

可用于：

1. 内部验证
2. 样本生成
3. 冷启动数据准备

不可直接定义为：

1. 正式产品主链路
