# EGoclaw App 与 UI 说明

## 1. 产品结构

EGoclaw 由两个独立部分组成：

1. 主应用窗口
2. 系统级桌宠灵宝

主应用窗口负责：

1. 展示收藏同步结果
2. 展示知识图谱
3. 展示当前意图
4. 展示当前动作
5. 展示内容蒸馏和解释

系统级桌宠灵宝负责：

1. 常驻桌面
2. 检测抖音是否在前台
3. 在触发条件满足时从抖音边缘跳出
4. 提醒用户
5. 点击后拉起主应用窗口

桌宠不是主应用里的一个浮层。
桌宠是独立窗口。

---

## 2. 当前 UI 问题

现在桌面之所以冗余，问题有三个：

1. 把系统状态、Agent、图谱、提醒、控制台同时堆在首页
2. 没有明确主次，用户第一眼不知道该看哪里
3. 把桌宠也放进了软件界面逻辑里，导致边界混乱

正确做法：

1. 桌宠极简
2. 首页只放当前最重要的信息
3. Agent 细节只在同步页或调试页出现
4. 图谱和蒸馏各自独立页面
5. 主应用负责深度查看，桌宠负责轻触发

---

## 3. UI 总原则

UI 风格目标：清晰、干净、克制，接近 Claude 的信息组织方式。

### 3.1 视觉原则

1. 浅色背景
2. 大留白
3. 少颜色
4. 弱边框
5. 卡片数量少
6. 每屏只保留一个主任务

### 3.2 信息原则

1. 首页只放“当前主意图”和“当前动作”
2. 二级信息放在下方或次级页面
3. 多 Agent 不默认铺开
4. 图谱不和蒸馏、设置、日志混放
5. 日志和调试信息只在设置或同步页中出现

### 3.3 交互原则

1. 用户打开 App 后，第一眼必须知道现在最值得开始什么
2. 用户点击桌宠后，必须直接进入当前动作或当前路径
3. 用户不需要在首页做复杂浏览
4. 首页不出现超过 4 个主模块

---

## 4. 桌宠 UI

桌宠是独立于主应用的系统级窗口。

### 4.1 桌宠待机态

表现：

1. 小火人本体
2. 默认停靠在桌面一角
3. 不显示大块面板

功能：

1. 常驻
2. 等待触发
3. 点击可打开主应用

### 4.2 桌宠触发态

触发条件：

1. 抖音在前台
2. 当前存在可触发意图
3. 当前时间和上下文满足提醒条件

表现：

1. 从抖音窗口边缘跳出
2. 跳到桌面可见区域
3. 弹出小气泡

提醒气泡只包含：

1. 一句提醒
2. 一个主按钮
3. 两个次按钮

按钮：

1. 现在开始
2. 晚点提醒
3. 不是这个

### 4.3 桌宠 UI 不能做什么

桌宠不应该：

1. 展示完整图谱
2. 展示多列面板
3. 展示复杂设置
4. 展示大量 Agent 信息

桌宠只做：

1. 触发
2. 提醒
3. 跳转

---

## 5. 主应用窗口 UI

主应用窗口只负责深度查看和执行。

推荐结构：

1. 左侧导航
2. 顶部标题区
3. 中央主内容区

页面风格：

1. 浅背景
2. 单层卡片
3. 标题明显
4. 次级描述弱化

---

## 6. 页面结构

主应用窗口保留以下页面：

1. 启动与连接页
2. 首页总览
3. 同步中心页
4. 成长地图页
5. 内容蒸馏页
6. 当前行动页
7. 对话与解释页
8. 设置页

---

## 7. 每页 UI 说明

## 7.1 启动与连接页

### 页面目标

完成连接和首次初始化。

### 页面结构

只保留三块：

1. 标题区
2. 连接按钮区
3. 初始化进度区

### 页面内容

标题区：

1. Logo
2. 一句话说明
3. 灵宝形象

连接按钮区：

1. 登录
2. 连接抖音

初始化进度区：

1. 收藏同步
2. 蒸馏中
3. 图谱生成中

这个页面不要出现：

1. 图谱
2. Agent 列表
3. 复杂说明文字

---

## 7.2 首页总览

### 页面目标

让用户第一眼知道：

1. 当前最值得开始什么
2. 为什么是这个
3. 下一步是什么

### 首页只保留 4 个模块

#### 1. 当前主意图卡

内容：

1. 当前意图名称
2. 一句话解释
3. 触发原因摘要

这是首页最大的卡片。

#### 2. 当前动作卡

内容：

1. 当前动作
2. 预计耗时
3. 为什么是现在
4. 开始按钮

#### 3. 热爱方向简表

内容：

1. Top 3 方向
2. 热度变化

不要在首页直接画复杂图谱。

#### 4. 最近一次灵宝提醒

内容：

1. 最近提醒内容
2. 触发原因
3. 用户反馈

### 首页不能出现

1. 整屏 Agent 面板
2. 大量日志
3. 大型图谱
4. 同步详情表

首页是“决策页”，不是“系统后台页”。

---

## 7.3 同步中心页

### 页面目标

展示数据进入系统的处理过程。

### 页面结构

1. 连接状态区
2. 同步任务区
3. 原始收藏区
4. Agent 处理链路区

### 这里才展示 Agent

Agent 只在这个页面出现：

1. Ingestion Agent
2. Distill Agent
3. Intent Agent
4. Graph Agent
5. Planner Agent
6. Nudge Agent
7. Memory Agent

展示方式：

1. 纵向流程
2. 每个 Agent 一张简卡
3. 只显示输入、输出、状态

不要做成大屏控制台。

---

## 7.4 成长地图页

### 页面目标

展示个人知识图谱。

### 页面结构

1. 图谱画布
2. 当前路径摘要
3. 节点详情抽屉

### UI 原则

1. 图谱居中
2. 默认只高亮当前路径
3. 其他路径弱化
4. 不要同时塞很多表格

### 右侧或底部详情区只显示

1. 节点名
2. 来源视频
3. 当前动作
4. 下一步

---

## 7.5 内容蒸馏页

### 页面目标

展示视频如何变成结构化 skill。

### 页面结构

1. 左侧视频列表
2. 右侧蒸馏结果

### 右侧蒸馏结果只显示

1. topic
2. skill
3. stage
4. friction
5. action
6. 证据链

这个页面应该像“结构化理解页”，不是素材墙。

---

## 7.6 当前行动页

### 页面目标

让用户立即开始。

### 页面结构

1. 当前动作卡
2. 行动拆解清单
3. 支撑视频列表

### 当前动作卡必须最突出

内容：

1. 动作名称
2. 预计耗时
3. 为什么是这一步
4. 开始按钮

### 行动拆解清单

只保留：

1. 3 到 5 个步骤

不要把它做成复杂项目管理工具。

---

## 7.7 对话与解释页

### 页面目标

承接用户和灵宝的解释性交互。

### 页面结构

1. 聊天气泡区
2. 快捷问题区
3. 当前解释区

### 这个页面应该回答

1. 为什么提醒我这个
2. 你是怎么判断的
3. 我今天最该开始什么
4. 这条路径从哪里来的

不要把它做成开放式大聊天产品。

---

## 7.8 设置页

### 页面目标

管理桌宠和同步行为。

### 页面结构

1. 账号设置
2. 提醒设置
3. 桌宠设置
4. 数据设置

### 设置页才允许出现

1. 调试信息
2. 运行日志
3. 手动同步按钮

---

## 8. 桌宠与主应用的关系

桌宠和主应用不是一套 UI。

关系是：

1. 桌宠负责触发
2. 主应用负责承接

具体链路：

1. 用户打开抖音
2. 后台检测抖音在前台
3. 判断当前意图可触发
4. 灵宝从抖音边缘跳出
5. 用户点击“现在开始”
6. 主应用打开到“当前行动页”

所以 UI 上必须区分：

1. 桌宠是极简触发层
2. App 是完整内容层

---

## 9. 第一版 UI 落地要求

第一版必须做到：

1. 首页只保留 4 个主要模块
2. 桌宠独立于主应用
3. 同步页才显示 Agent
4. 图谱页单独展示图谱
5. 当前行动页单独承接执行

第一版不要再做：

1. 大杂烩首页
2. 桌宠嵌在软件里
3. 一屏同时展示图谱、Agent、日志、设置

---

## 10. 最终 UI 关键词

关键词：

1. 清晰
2. 干净
3. 克制
4. 少块
5. 强主次
6. 桌宠独立
7. 首页只看“现在最该开始什么”
